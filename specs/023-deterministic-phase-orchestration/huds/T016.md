# HUD: T016 — Tighten lock/retry sequencing around approval-gated execution in `scripts/pipeline_driver_state.py`

## Working Memory

**File:Symbol**: `scripts/pipeline_driver_state.py:acquire_feature_lock`  
**Callers**: `tests/unit/test_pipeline_driver.py:test_acquire_feature_lock_blocks_active_other_owner`, `tests/unit/test_pipeline_driver.py:test_acquire_feature_lock_replaces_stale_owner`  
**Reuse path**: Reuse the existing lock record format and only refine retry semantics for the same owner.

## Solution Sketch

Treat a same-owner reacquire after expiry as a retry reuse instead of a takeover. Keep other-owner stale replacement behavior intact, so the lock file still resolves ownership deterministically while giving the approved actor a stable retry path.

## Test Specification

Assert that the same owner reacquiring after expiry returns `acquired=True`, `reused=True`, `stale_replaced=False`, and `reason=stale_lock_reused`, while a different owner still receives stale replacement or hold rejection as appropriate.

## Security Review

No new security surface is added. The change only clarifies existing lock ownership semantics.
