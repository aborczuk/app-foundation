# HUD: T014 — Require explicit approval token before side-effectful phase execution in `scripts/pipeline_driver.py`

## Working Memory

**File:Symbol**: `scripts/pipeline_driver.py:main` and `scripts/pipeline_driver.py:enforce_approval_breakpoint`  
**Callers**: `tests/integration/test_pipeline_driver_feature_flow.py:test_approval_breakpoint_blocks_without_token`, `tests/unit/test_pipeline_driver.py:test_main_generative_route_executes_handoff_adapter`  
**Reuse path**: Reuse the existing breakpoint helper and keep the approval gate scoped to `implement` only.

## Solution Sketch

Add an explicit breakpoint config in `main()` only for the `implement` phase, then call `enforce_approval_breakpoint()` before dispatch. Keep the gate deterministic: missing or wrong-scope tokens return the blocked envelope, and `run_step`/append paths stay cold when approval is absent.

## Test Specification

Assert that missing and invalid approval tokens both return exit code `1`, emit `approval_required`, and never reach `run_step` or ledger append. Also assert that the breakpoint config passed to the helper targets `implement` with `required_scope=security_sensitive_migration`.

## Security Review

Approval is explicitly required before side effects. The token is only interpreted as a scope prefix, and the gate stays outside the execution payload path.

## Functional Goal

**Story Goal**: User Story 2 - Approval-Gated Side-Effect Execution  
**Acceptance Criteria**: Denied or mismatched approval blocks execution with zero side effects; valid approval allows the runtime path to proceed.

## Quality Guards

- Keep approval boundaries explicit and human-visible.
- Do not leak tokens into logs or status output.
- Preserve deterministic blocked envelopes for denied approval.
- Domains 13, 14, and 17 remain in scope.
