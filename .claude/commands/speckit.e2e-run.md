## User Input

```text
$ARGUMENTS
```

You **MUST** consider the user input before proceeding (if not empty).

## Purpose

This command **executes** the E2E testing pipeline generated by `/speckit.e2e`. It runs the pipeline script, walks the user through required human-assisted checks, and validates results. It is the E2E equivalent of `/speckit.checkpoint` — a hard validation gate.

**This command can be invoked in three ways:**
1. **Standalone**: `/speckit.e2e-run` — runs the full pipeline
   - Feature purpose: carry the one-line feature purpose from `spec.md` through this step.
2. **With a section**: `/speckit.e2e-run Section 2` or `/speckit.e2e-run US1` — runs a specific user story section
   - Feature purpose: carry the one-line feature purpose from `spec.md` through this step.
3. **From `/speckit.implement`**: Called automatically after a user story phase completes
   - Feature purpose: carry the one-line feature purpose from `spec.md` through this step.

## Outline

1. **Locate the feature and E2E artifacts**:
   - Feature purpose: carry the one-line feature purpose from `spec.md` through this step.
   - Run `.specify/scripts/bash/check-prerequisites.sh --json` from repo root and parse FEATURE_DIR and AVAILABLE_DOCS list. All paths must be absolute.
   - Read `FEATURE_DIR/e2e.md` — if it does not exist, **STOP** and suggest running `/speckit.e2e` first.
   - Locate the E2E script: search `scripts/e2e_*.sh` for a script matching this feature. If multiple exist, use the one referenced in e2e.md. If none exist, **STOP** and suggest running `/speckit.e2e` first.
   - Read plan.md for tech stack context (how to run the software, config paths).

2. **Determine which sections to run**:
   - Feature purpose: carry the one-line feature purpose from `spec.md` through this step.
   - If user input specifies a section (e.g., "Section 2", "US1", "preflight"): run that specific section.
   - If user input is empty or "full": run all sections in order (preflight → per-story → final).
   - If user input is "preflight": run only the preflight/dry-run section.
   - If user input is "verify": run only the verification/check section.
   - If called from `/speckit.implement` with a phase number: map the phase to its corresponding E2E section (e.g., Phase 3 = User Story 1 section).

3. **Check prerequisites**:
   - Feature purpose: carry the one-line feature purpose from `spec.md` through this step.
   - Parse the prerequisites section of e2e.md.
   - For each prerequisite, verify availability:
     - **Config files**: Check if they exist at the expected path.
     - **Env vars**: Check if they are set (do NOT log their values).
     - **External services**: Attempt a lightweight connectivity check if possible (e.g., check if a port is listening), or ask the user.
   - **If any prerequisite is missing**:
     - List all missing prerequisites clearly.
     - Report the run as **BLOCKED (prerequisites)** and stop.
     - Instruct the user to satisfy prerequisites and re-run. Blocked/skipped is not an acceptable completion state.

4. **Execute the pipeline**:
   - Feature purpose: carry the one-line feature purpose from `spec.md` through this step.

   **For each section being run:**

   a. **Announce the section**: Display the section name, purpose, and what will be validated.

   b. **Run user asks** (human-gated prerequisites for this section):
      - Display the checklist of things the user must confirm before this section.
      - **Ask the user** to confirm each item. Format:
        ```
        Before running [Section Name], please confirm:
        - [ ] [External service] is running and accessible
        - [ ] [Config] is set up correctly
        Do you want to proceed? (yes/skip/stop)
        ```
      - If user says "skip": mark section as FAIL (not validated) and halt.
      - If user says "stop": halt the entire pipeline.
      - If user says "yes" or "proceed": continue.

   c. **Run automated checks**:
      - Execute the E2E script for this section using the appropriate subcommand.
      - Capture stdout, stderr, and exit code.
      - Parse output for pass/fail indicators.
      - If the script uses interactive prompts (`read -r -p`), relay these to the user and pass their responses.
      - **Async process management guard** (mandatory for async/event-loop/background-worker sections):
        - Verify lifecycle evidence exists for spawned tasks/processes (start, readiness, shutdown).
        - Fail the section if loop/lifecycle errors appear (e.g., `"event loop is already running"`, pending-task destruction warnings).
        - Fail the section if cleanup checks indicate orphan processes/tasks after execution.
      - **State safety and reconciliation guard** (mandatory when local state mirrors live external state):
        - Verify reconciliation evidence exists before side-effect decisions in the section path.
        - Fail the section if source-of-truth drift leaves local records active.
        - Fail the section if state transitions are logged without corresponding persisted lifecycle updates.
      - **Local DB transaction integrity guard** (mandatory when local DB mutations represent lifecycle/risk/financial state):
        - Verify transaction-boundary evidence exists for multi-step writes in the section path.
        - Fail the section if partial local writes or impossible lifecycle transitions persist after failure/retry paths.
        - Fail the section if persistence exceptions are swallowed or commit/rollback outcomes are ambiguous.

   d. **Run human verification steps**:
      - For any step marked as "Human verify" in e2e.md, present the verification criteria to the user.
      - **Ask the user**: "Please verify: [criteria]. Did this pass? (yes/no)"
      - Record their response.

   e. **Record section result**:
      - **PASS**: All automated checks passed, async lifecycle guard passed (if applicable), state safety guard passed (if applicable), local DB transaction guard passed (if applicable), AND all human verifications confirmed.
      - **FAIL**: Any automated check failed OR any human verification denied.

5. **Build the results report**:
   - Feature purpose: carry the one-line feature purpose from `spec.md` through this step.

   ```text
   ## E2E Pipeline Results: [Feature Name]

   **Date**: [ISO date]
   **Config**: [config path used]
   **Scope**: [full | section N | preflight]

   | Section | Status | Details |
   |---------|--------|---------|
   | Preflight | PASS | Dry-run completed, 3 cycles observed |
   | US1: [Title] | PASS | order_filled verified, DB records confirmed |
   | US2: [Title] | FAIL | Prerequisite missing — IBKR Gateway unavailable |
   | Full Feature | FAIL | Cross-story integration check not satisfied |

   **Overall**: FAIL (2 passed, 2 failed)
   ```

6. **Determine next action based on results**:
   - Feature purpose: carry the one-line feature purpose from `spec.md` through this step.

   - **If ALL sections PASS**:
     - Report: "E2E Pipeline PASSED"
     - If called from `/speckit.implement`: return PASS status so implementation can proceed.

   - **If ANY section FAILS**:
     - Report the full results table with failure details.
     - Analyze each failure:
       - Is it a code bug? (unexpected behavior, wrong output)
       - Is it an environment issue? (service down, config wrong)
       - Is it a test plan issue? (e2e.md expectations are wrong)
     - **Suggest specific fixes** for each failure.
     - If called from `/speckit.implement`: return FAIL status. The phase is NOT complete.

   - **If any section is SKIPPED or BLOCKED**:
     - Treat the run as FAIL for acceptance purposes.
     - Report what must be fixed before re-run.
     - If called from `/speckit.implement`: return FAIL status. The phase is NOT complete.

7. **Log the run result**:
   - Feature purpose: carry the one-line feature purpose from `spec.md` through this step.
   - Append a run summary to `FEATURE_DIR/e2e.md` as a comment block at the bottom:

     ```markdown
     <!-- E2E Run: PASS | 2026-03-12 | full | 3/3 sections passed -->
     ```

   - If the run was blocked or had failures:

     ```markdown
     <!-- E2E Run: FAIL | 2026-03-12 | full | prerequisites blocked or section failure -->
     ```

## Execution Modes

### Preflight Mode (`/speckit.e2e-run preflight`)

Runs only the dry-run smoke test. No external dependencies needed beyond the application itself. This is the cheapest E2E check and should be run frequently, but it is not sufficient for feature acceptance.

### Section Mode (`/speckit.e2e-run US1` or `/speckit.e2e-run Section 2`)

Runs a single user story's E2E section. Useful after completing a specific story during implementation.

### Full Mode (`/speckit.e2e-run` or `/speckit.e2e-run full`)

Runs the entire pipeline end-to-end: preflight → all per-story sections → final full-feature section. This is the most thorough validation and should be run after the feature is complete and after significant changes.

### Verify Mode (`/speckit.e2e-run verify`)

Runs only the lightweight verification checks (log tails, DB queries, state inspection). No interactive steps. Useful for quick state checks, but not sufficient for feature acceptance.

## Notes

- This command **executes** the pipeline — it does not generate it. Use `/speckit.e2e` to generate or regenerate E2E artifacts.
- The command relays interactive prompts from the E2E script for non-automatable external checks.
- Async/event-loop sections are not passable if lifecycle cleanup fails or loop-management errors appear in logs/output.
- Stateful integration sections are not passable if live-vs-local drift remains unresolved.
- Local DB mutation sections are not passable if partial writes, impossible lifecycle transitions, or ambiguous transaction outcomes are detected.
- Preflight mode is safe to run anytime — it uses `--dry-run` or equivalent and produces no side effects.
- Full mode requires human time and external dependencies. Budget accordingly.
- Results are idempotent — running the same section twice overwrites the previous result comment in e2e.md.
- When called from `/speckit.implement`, the implement workflow should pass the section identifier and await the result before proceeding to the next phase.
- Acceptance rule: if E2E does not PASS, the feature/phase is not done.
